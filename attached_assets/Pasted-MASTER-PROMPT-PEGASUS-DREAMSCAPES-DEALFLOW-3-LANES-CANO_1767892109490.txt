MASTER PROMPT — PEGASUS DREAMSCAPES DEALFLOW (3 LANES + CANONICAL FORMS + OFFER STUDIO + NEGOTIATION ROOM)

We need to refactor DealFlow so it is organized, consistent, and “money-saving” by eliminating duplicated forms and confusing flows. Right now the same user action opens different forms depending on view (Grid/Swipe/View Deal). That must be fixed.

CORE REQUIREMENT (NON-NEGOTIABLE)

Every deal lane must have ONE canonical set of forms + ONE canonical routing flow, reused everywhere:

Grid view buttons

Swipe view buttons

View Deal / View Details page buttons

Any “Take Action” panel buttons

Negotiation Room actions

No more “different form depending on where you clicked.”

1) DealFlow must have 3 lanes (tabs)
A) WHOLESALES (Wholesale Assignments)

Purpose: buy/assign contract OR JV with wholesaler. Negotiation is simple and fast.

Primary actions on wholesale cards:

View Deal

Accept Terms (simple)

Counter Offer (simple)

Request JV Partnership (simple)

B) DEVELOPMENTS / CAPITAL RAISES

Purpose: invest in an operator project (Debt/Equity/Hybrid). Negotiation is advanced.

Primary actions on capital cards:

View Details

Accept Terms

Counter Offer

Go to Negotiation Room (if already in progress)

C) LISTINGS (On-market + Off-market + Move-in Ready)

Purpose: request info / schedule showing / submit interest. Not a negotiation-heavy flow.
Label badges: On Market / Off Market / Move-In Ready / Featured (and optional “Needs Renovation” or “Minor Updates”)

Primary actions on listing cards:

View Listing

Request Info

Schedule Showing
(Optional: “Submit Offer” can exist later, but for now it’s inquiry-focused.)

2) Canonical Form System (fix the inconsistency issue)

We must implement a single “Form Router” layer so ALL entry points open the same forms.

Build a canonical action router:

Create a single function like:
openDealAction({ laneType, actionType, dealId, context })

laneType: wholesale | capital | listing

actionType: accept | counter | jv_request | invest_accept | invest_counter | request_info | schedule_showing

context: grid | swipe | detail | negotiation_room

This router decides which canonical component to open.

Canonical forms to implement (single-source components)

WHOLESALE

WholesaleAcceptTermsModal

WholesaleCounterOfferModal

WholesaleJVRequestModal

CAPITAL
4) CapitalAcceptTermsModal (fast accept path)

LISTINGS

Rule: Grid/Swipe/Detail cannot create their own forms. They ONLY call the router.

3) Offer Studio Rules (Capital Raises ONLY)

Offer Studio is the “drafting room” and must be used for:
✅ Capital project counter offers (advanced structuring)

Offer Studio must NOT be used for:
❌ Wholesale
❌ Listings
❌ Capital “accept as posted” (should be fast accept)

Capital Flow

When user clicks “Make Offer / Invest” on a Capital Raise:
Step 1: show an intent choice:

Accept Terms (fast path)

Counter Terms (Offer Studio path)

If Accept Terms (FAST PATH)

Open CapitalAcceptTermsModal with:

Investment Amount (min enforced)

Acknowledgements checkbox(es)

Optional message
Submit → creates Offer record with type='ACCEPT' and snapshots operator terms
Then route to Negotiation Room

If Counter Terms (FULL PATH)

Open Offer Studio:
Steps:

Deal Structure (Debt / Equity / Hybrid)

Terms (return %, split, term length, payment schedule, protections)

Funding / Notes

Review & Submit

Submit → creates Offer record with type='COUNTER' and snapshots terms
Then route to Negotiation Room

4) Negotiation Room = Escrow Workspace (applies to Wholesale + Capital)

Negotiation Room should feel like escrow: it opens once 2 principals are negotiating.

Negotiation Room must be unified + consistent:

Wholesale negotiation room uses simple offer structure (price/fee, EMD, close date, inspection, notes)

Capital negotiation room references Offer Studio offers (debt/equity/hybrid structure)

Layout requirement (3 columns)

Column A: Chat

Operator/Wholesaler ↔ Buyer/Investor

System messages (offer submitted, accepted, countered)

Column B: Offer Ladder / History

Every offer version listed chronologically with status + terms snapshot

Actions:

Accept

Reject

Counter (opens the correct canonical form)

Wholesale counter opens WholesaleCounterOfferModal

Capital counter opens Offer Studio prefilled from last offer

Column C: Peggy AI Advisor

Reads current deal + latest offer

Shows:

Risk flags

Fairness analysis

Suggested counter ranges

Quick actions (suggest optimal terms, simplify terms, protections)

Critical data rule: Snapshot terms

When an offer is created, store a snapshot of:

the terms at that moment

who proposed it

timestamp
so the ladder never changes even if the original deal is edited later.
