REPLIT — PLEASE READ CAREFULLY. This is a product architecture + UX spec.

We need MarketFlow to support THREE different marketplace lanes with THREE different deal types. The main problem right now is form duplication + inconsistent flows across Grid vs Swipe vs View Deal. FIX MUST BE ARCHITECTURAL: forms must be tied to dealType (data), not tied to view (UI).

========================================
A) CORE CONCEPT: 3 MARKET LANES / 3 DEAL TYPES
========================================

MarketFlow has 3 sections (tabs) at the top:

1) WHOLESALES (Wholesale Assignments)
2) DEVELOPMENTS / CAPITAL RAISES
3) LISTINGS (Ready-to-move-in properties)

Every card/listing/project MUST have a required field:
dealType ENUM = "WHOLESALE_ASSIGNMENT" | "CAPITAL_RAISE" | "LISTING"

This dealType drives:
- which buttons show on the card
- which form opens
- which negotiation room schema is used
- which fields exist

Non-negotiable: The SAME dealType must open the SAME form, regardless of entry point:
- Grid view card buttons
- Swipe view card buttons
- View Deal page buttons
- Negotiation Room buttons

Views are just different presentations. They must not create different forms or logic.

========================================
B) SINGLE ACTION ROUTER (MANDATORY)
========================================

Implement a single “Deal Action Router” used by ALL buttons:

openDealAction({
  dealId,
  dealType,
  action: "VIEW" | "ACCEPT" | "COUNTER" | "OFFER" | "JV" | "INVEST" | "INQUIRE" | "SCHEDULE" | "NEGOTIATION_ROOM"
})

ALL buttons in ALL views must call openDealAction(). No view-specific modals.

openDealAction() decides which UI to open:
- modal component
- or route page
based on dealType + action.

========================================
C) FORMS: CANONICAL FORMS ONLY (NO DUPLICATES)
========================================

We will have EXACTLY these canonical forms:

----------------------------------------
FORM 1 — WholesaleAssignmentOfferModal (SIMPLE)
----------------------------------------
DealType: WHOLESALE_ASSIGNMENT
Used for: "Offer" and "Counter" on wholesale assignment deals.
This is NOT Offer Studio. Keep it simple and fast.

Fields (required unless noted):
- assignmentFeeOffer (currency) (default to listedAssignmentFee)
- earnestMoney (currency) (default sensible value)
- closingDate (date)
- inspectionDays (number)
- messageToWholesaler (text, optional)

Context header (read-only):
- contractPrice
- ARV
- repairs
- listedAssignmentFee
- negotiableFlag

Actions:
- Submit Offer (or Submit Counter if this is a counter)
- Cancel
- (optional link) “Continue in Negotiation Room” BUT it must still use SAME data schema.

Rules:
- No “Deal Structure” step.
- No financing tabs.
- No equity/hybrid cards.
- No investor terms.
- Must look and behave the same from Grid/Swipe/View Deal.

----------------------------------------
FORM 2 — WholesaleJVRequestModal (SIMPLE)
----------------------------------------
DealType: WHOLESALE_ASSIGNMENT
Used for: wholesaler-to-wholesaler JV requests.

Fields:
- proposedSplitPercent (e.g., 50/50)
- whoBringsBuyer (Me / You / Both)
- expectedAssignmentFee (optional)
- messageToWholesaler (text)

Actions:
- Submit JV Request
- Cancel

Rules:
- Remove “Investment Strategy” and “Funding Source” from JV form. Those do not belong here.

----------------------------------------
FORM 3 — OfferStudio (ADVANCED) — Capital Only
----------------------------------------
DealType: CAPITAL_RAISE
Offer Studio MUST ONLY be used for capital raise deals, NOT wholesales.

Offer Studio steps can remain:
Step 1: Choose structure (Debt / Equity / Hybrid)
Step 2: Terms & Price (investment terms)
Step 3: Financing (if needed, only for capital context)
Step 4: Review & Submit

But its fields must be capital-focused:
- investmentAmount
- expectedReturn (interest/preferred return)
- profitSplit (investor share)
- termLengthMonths
- optional notes / conditions
- optional milestones (later)

Header must show:
- fundingTarget
- raisedSoFar
- remaining
- minInvestment
- structure type
- progress bar (raised/target)

Actions:
- Accept Terms (1-click with operator terms)
- Counter Terms (submits investor proposed terms)
- Message operator / go to negotiation room

----------------------------------------
FORM 4 — ListingInquiryModal (LISTINGS ONLY)
----------------------------------------
DealType: LISTING
Listings are “Ready-to-move-in properties” (on-market/off-market). This is NOT negotiation.
Goal is inquiry + showing scheduling.

Fields:
- fullName
- email
- phone
- interestType dropdown: (Request Info, Schedule Showing, Ask About Offer, Ask About Financing)
- preferredShowingDates (2-3 options OR date/time picker)
- preApproval toggle (optional)
- message

Listing card must show tags:
- On Market / Off Market

Actions:
- Send Inquiry
- Cancel

Rules:
- No negotiation room required (at least not in Phase 1).
- No assignment fee.
- No capital investment terms.

========================================
D) NEGOTIATION ROOM MUST BE TYPE-AWARE
========================================

NegotiationRoom must support 2 schemas:
1) Wholesale negotiation
2) Capital raise negotiation

If dealType == WHOLESALE_ASSIGNMENT:
- show Offer Ladder for assignment offers/counters
- show Chat
- offers include: assignmentFee, earnest, closeDate, inspectionDays, message
- Accept / Reject / Quick Counter must open WholesaleAssignmentOfferModal (same canonical form)

If dealType == CAPITAL_RAISE:
- show Terms (operator terms + progress bar)
- show Offer Ladder (investment commitments/counters)
- show Messages
- Accept / Counter opens OfferStudio (same canonical form)
- must display raised/target progress

No mixed schemas.

========================================
E) MARKETFLOW UI REQUIREMENTS
========================================

1) Top-level MarketFlow page must have 3 tabs:
- Wholesale Assignments
- Capital Raises
- Listings (NEW — missing now)

2) Each tab supports Grid and Swipe view toggles (fine).
But again: Grid and Swipe must call the SAME openDealAction and the SAME forms.

3) Cards must show correct CTAs based on dealType:

WHOLESALE_ASSIGNMENT card CTAs:
- View Deal
- Make Assignment Offer (or Accept Terms if you want that label)
- Counter Offer

CAPITAL_RAISE card CTAs:
- View Details

- Counter Terms

LISTING card CTAs:
- View Listing
- Request Info
- Schedule Showing

4) Labels must be consistent:
- Wholesale uses: “Assignment Offer” / “Assignment Fee”
- Capital uses: “Investment Amount” / “Terms”
- Listing uses: “Showing” / “Inquiry”

========================================
F) DATA INTEGRITY (WHY SOME SECTIONS ARE NOT SHOWING)
========================================

Right now fields are missing in some views because they’re passing partial objects.
Fix: ALL forms fetch full deal context by dealId when opened.

Implement:
GET /api/deals/{dealId}/context
returns:
- deal core fields
- dealType
- lane (wholesale/capital/listing)
- operator terms (if capital)
- wholesale terms (if wholesale)
- listing details (if listing)
- existing offers/counters
- permissions for current user

All modals/forms must load from this endpoint so every view shows complete sections.

========================================
G) ACCEPT TERMS VS COUNTER — BEHAVIOR
========================================

WHOLESALE:
- “Accept Terms” should mean “Accept listed assignment fee and standard dates”
  -> opens WholesaleAssignmentOfferModal prefilled (assignmentFeeOffer=listedAssignmentFee, standard earnest/date), user confirms and submits.
- “Counter Offer” opens the same modal but flagged as counter.

CAPITAL:
- “Accept Terms” should mean investor accepts operator terms with chosen investment amount (>= min)
  -> either quick modal OR OfferStudio prefilled. Either way, it must store the same schema as a commitment.
- “Counter Terms” opens OfferStudio with editable expectedReturn/profitSplit/term/etc.

LISTING:
- No accept/counter. Only inquiry/schedule.
