# Website Experience Blueprint v1.0

> **Superseded by `website-marketflow-blueprint-v1.3.1.md` for the public website + MarketFlow scope.** This v1.0 doc remains as historical reference for sections v1.3.1 does not restate.

## Purpose

This blueprint locks the public Pegasus Dreamscapes website experience before further voice, differentiation, and polish work. It is the governing document for the marketing website: what the website says, how it routes users, what each page is responsible for, where compliance language belongs, and where the public website ends and HQ / MarketFlow begins.

The public website should feel like a strategy-first real estate operating company, not a generic investor, brokerage, wholesaler, or cash-buyer site. The core positioning is:

- Pegasus Dreamscapes Corp is a strategy-first real estate operating company.
- The operating identity is "The Deal Architect."
- The promise is "Where others see impossible, we see a path."
- The doctrine is "Every property gets a path."
- The system routes opportunities through strategy, structure, execution, and lawful next steps.

## Homepage Order

The homepage is locked to this strategic order:

1. Hero.
2. Every Property Gets A Path.
3. Services / Three Pillars.
4. Development Pathway.
5. Strategy Stack and Structure Stack.
6. Outcome Lanes.
7. Featured Project.
8. MarketFlow Beta.
9. Operating Principles.
10. Final CTA.

Do not add extra homepage sections without a clear job that strengthens this sequence. Do not reintroduce generic trust-logo, fake-stat, newsletter, or cash-buyer style blocks into the homepage.

## Homepage Jobs

### Hero

The hero must establish the brand and doctrine immediately:

- Pegasus Dreamscapes.
- The Deal Architect.
- Development, Investments, Systems.
- "Where others see impossible, we see a path."
- "Dream it. Build it. Live it."
- Primary CTA: submit a property or start a routed inquiry.
- Secondary CTA: move deeper into the Development Pathway or operating doctrine.

The hero should be cinematic and premium, but still clear. It should not become a vague luxury real estate splash page.

### Every Property Gets A Path

This section explains the strategy-review model:

- What is the property?
- What is the situation?
- What does the owner want?
- What is the best economic path?
- What is the safest structure?
- Should Pegasus participate?

This is the first major differentiation block. It must avoid "we buy houses" commodity language.

### Services / Three Pillars

The three pillars are:

- Development.
- Investments.
- Systems.

These are not three unrelated services. They are one operating company stack. Development is the spine; Investments and Systems support execution.

### Development Pathway

The four-phase pathway must stay disciplined:

- Phase 1: Today - ADU additions, forced-value rehabs, fix-and-flips, BRRRR acquisitions, small-scale development.
- Phase 2: Next - small multi-unit conversions, 2 to 4 unit projects, co-developer relationships.
- Phase 3: Growth - ground-up infill construction on Pegasus-controlled lots, BuildForge-supported project management, multiple active projects.
- Phase 4: Legacy - master-planned classical neighborhoods and larger-scale development.

Phases 2 through 4 are trajectory, not current capability claims. Keep the "earned, not assumed" disclaimer.

### Strategy And Structure Stacks

The website should separate strategy from structure:

- Strategy examples: fix and flip, BRRRR / buy and hold, ADU / development, wholesale / assignment.
- Structure examples: direct acquisition, JV / co-GP, creative finance, referral / listing.

The user should understand that Pegasus does not force every opportunity into a single offer type.

### Outcome Lanes

The outcome-lane model is a core differentiator. The primary lanes include:

- Pegasus Acquisition.
- Wholesale Assignment.
- JV / Partnership.
- KW Listing Lane.

Secondary lanes include:

- Referral Lane.
- MarketFlow Distribution.
- Incubation / Nurture.
- Archive Intelligence.

The lane model should be described as disciplined routing, not guaranteed monetization.

### Featured Project

The project section should prove execution without overclaiming scale. Show a real or representative case study with:

- Asset type.
- Starting condition.
- Scope.
- Strategy.
- Outcome.
- Lessons or repeatable operating insight.

### MarketFlow Beta

MarketFlow should be positioned as private-beta infrastructure behind the public website:

- Invite-only network.
- Private deal-flow routing.
- Role-aware workflows.
- Operational system, not a public marketplace free-for-all.

### Operating Principles

Operating principles should reinforce doctrine:

- Strategy before speed.
- Structure before slogans.
- Execution before hype.
- Every lead is reviewed.
- Not every property gets an offer.

### Final CTA

The final CTA should route clearly:

- Submit a property.
- Invest / partner.
- Contact Pegasus.

It should not introduce a new concept at the end of the page.

## Role-Mode Router

The public website must route users by intent:

- Seller: `/sell`, property review, outcome routing, direct contact.
- Investor / capital partner: `/invest`, private inquiry, non-public-offering language.
- Wholesaler / operator: `/submit-deal`, deal submission, MarketFlow pathway.
- Buyer: `/buyers` or `/buy`, buyer opportunities and intake.
- General contact: `/contact`, routed inquiry.
- Existing network member: `/marketflow`, authenticated or demo-aware portal.

The website should help users self-select. It should not make every visitor go through the same CTA.

## Page Jobs

### `/sell`

Job: convert property owners and referral sources into serious property-review submissions.

Required content:

- Explain that Pegasus reviews the situation, not just the address.
- Show outcome-routing lanes.
- Preserve direct founder contact.
- Include compliance language that no offer is guaranteed.

### `/invest`

Job: qualify private capital, JV, and partner interest without implying a public securities offering.

Required content:

- Position capital as structure-first and relationship-driven.
- Explain debt, equity, and JV concepts carefully.
- Avoid performance guarantees, return promises, and public solicitation language.
- Include direct inquiry CTA and compliance language.

### `/projects`

Job: provide proof of execution and taste level.

Required content:

- Case-study cards.
- Clear strategy / status filtering if available.
- No fake volume claims.

### `/projects/:slug`

Job: turn each project into an execution narrative.

Required content:

- Situation.
- Before / after.
- Scope.
- Economics or strategic framing where available.
- Next project or conversation path.

### `/contact`

Job: route the user to the correct conversation.

Required content:

- Seller inquiry.
- Partner / investor inquiry.
- General / operational inquiry.
- Direct email, phone, and Bay Area location.

### `/about`

Job: explain the founder-led operating thesis.

Required content:

- Paolo "Apollo" Duran as founder and principal.
- Strategy-first positioning.
- The reason Pegasus is different from generic real estate brands.

### `/services`

Job: expand the three-pillar operating model.

Required content:

- Development.
- Investments.
- Systems.
- How the pillars support one another.

### `/submit-deal`

Job: collect deal submissions from wholesalers, operators, and referral partners.

Required content:

- Required deal facts.
- Lane expectations.
- No automatic acceptance language.

## CTA Paths

Primary CTAs should map to user intent:

- "Submit a Property" -> `/sell`.
- "Invest With Us" or "Partner Inquiry" -> `/invest`.
- "Submit a Deal" -> `/submit-deal`.
- "Enter MarketFlow" -> `/marketflow`.
- "Contact" -> `/contact`.
- "See Projects" -> `/projects`.

Avoid CTA sprawl. Every CTA should have a clear job and destination.

## Visual Rules From PR #6

The public website visual baseline is:

- Premium editorial real estate system.
- Deep navy, rich copper, warm cream, charcoal.
- Cormorant Garamond for editorial display headlines.
- Cinzel / display treatment for brand and carved-label moments.
- Montserrat / supporting caps for restrained kicker labels.
- Inter for body and functional UI.
- Cinematic image-led heroes.
- Hairline dividers, architectural strips, restrained borders.
- Bare icons or quiet icon treatment, not large decorative icon bubbles.
- Cards only where they frame real repeated items or tools.
- No fake trust logos.
- No fake stats.
- No commodity "cash buyer" visual language.
- No generic stock-business feel.

The visual system should feel disciplined and founder-led, not over-decorated. Decorative gradients, glows, and glass effects should be used only when they improve readability or hierarchy.

## Copy Guardrails

Use language that owns the Pegasus positioning:

- strategy-first operating company.
- The Deal Architect.
- every property gets a path.
- lanes.
- routing.
- structure.
- execution.
- development is the spine.
- private network.
- disciplined review.
- lawful next path.

Avoid language that weakens the brand:

- "We buy houses fast."
- "Guaranteed offer."
- "Highest price guaranteed."
- "Passive income guaranteed."
- "Public investment opportunity."
- "Marketplace for everyone."
- "Trusted by thousands" unless proven.
- Any fake volume, return, rating, certification, DRE, BBB, or licensing claim.

Use direct sentences. Avoid filler hype. Remove unnecessary em dashes during polish passes unless the punctuation is doing real work.

## Compliance Placement

Compliance language must be visible where decisions happen:

- Footer global disclosure.
- `/sell` near property-review and submission CTAs.
- `/invest` near capital / partner inquiry CTAs.
- `/submit-deal` near deal submission.
- Project pages where economics or outcomes are discussed.
- MarketFlow entry points where private-network access is described.

Required themes:

- Information is for general purposes only.
- No guarantee of offer, acceptance, funding, profit, or return.
- Nothing is an offer to buy or sell securities, real property, or investment products.
- Investment discussions are private, relationship-driven, and subject to proper review.
- Real estate outcomes depend on diligence, condition, market, title, financing, legal review, and execution risk.

Compliance language should be clear, calm, and readable. Do not hide it in tiny, low-contrast text.

## Website vs HQ Ownership

### Public Website Owns

- Brand positioning.
- Public education.
- Seller intake.
- Investor / partner inquiry.
- Project proof.
- Contact routing.
- MarketFlow preview and entry.
- Public SEO and content pages.

### HQ / MarketFlow Owns

- Authenticated user workflows.
- Role dashboards.
- Deal browsing.
- Negotiation rooms.
- Internal admin operations.
- Messaging.
- Notifications.
- Offer studio.
- Deal analytics.
- Document workflows.
- Private network activity.

Public pages may link into HQ / MarketFlow, but should not duplicate portal workflows. HQ / MarketFlow may use public brand language, but should not redefine the website doctrine.

## QA Checklist

Before merging website experience changes:

- Homepage order still matches this blueprint.
- Each public route has one clear job.
- CTAs route by user intent.
- Seller, investor, wholesaler, buyer, and general-contact paths are distinct.
- Development Pathway does not overclaim current scale.
- MarketFlow is framed as private beta / private network infrastructure.
- No fake stats, fake logos, fake certifications, or unverifiable trust claims.
- `/invest` avoids public-offering and guaranteed-return language.
- Compliance language is present near high-intent forms and in the footer.
- Visuals preserve PR #6 baseline.
- Copy uses Pegasus differentiators: strategy, routing, lanes, structure, execution.
- Website and HQ ownership remain separate.
- `npx tsc --noEmit` passes.
- `npm run build` passes.
